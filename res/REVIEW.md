# Code Review

## A. Introduction

This review explains what I learned while using and extending the calendar system, especially when I added the dashboard analytics feature. The system shows good architectural thinking with MVC structure which made navigation easier. But at the same time, some design and implementation choices made certain extensions harder than expected. My goal is to give balanced feedback with concrete examples so that each point is clear.

## B. Design Critique

The design has several good points. MVC separation is mostly clean, and the command-and-parser pattern made adding dashboard command straightforward. When I implemented the dashboard feature, I could simply create a new ShowCalendarDashboardCommand class following the same pattern as existing commands. Using interfaces also helps with abstraction and made testing easier. Parameter objects for event creation like EventCreationRequest help avoid mistakes with long parameter lists, so keeping them makes sense.

However, the design also has problems. There are too many small command classes which increases cognitive load when navigating the codebase. The controller layer has multiple handler classes like EventOperationsHandler and CalendarOperationsHandler that add extra indirection, making responsibilities unclear. Calendar and EventRepository classes also overlap in purpose, which can be confusing when tracing through the logic to understand how events are stored and retrieved.

There are clear design smells present. First, there is duplicate event-editing logic between Calendar and EventEditor classes. Calendar class has a private updateEventProperty method that uses switch statement, while EventEditor uses Map-based approach for the same purpose. When I needed to understand property editing for dashboard feature, I found similar logic in two places with different implementations, which violates DRY principle. To refactor this, Calendar should delegate all property editing to EventEditor class. Remove the updateEventProperty method from Calendar entirely and call EventEditor.updateProperty instead. This will eliminate duplication and ensure all property updates use consistent logic.

Second, there is Large Class smell in GUI components. CreateEventDialog is 693 lines, EditEventDialog is 531 lines, and SearchEventsDialog is 603 lines. These classes mix layout creation, event handling, validation logic, and business decisions. For example, EditEventDialog contains not only UI setup code but also logic to determine whether to edit single event or series, which belongs in controller layer. To refactor this, extract smaller dialog components like RecurringOptionsPanel and PropertyEditorPanel. Move the decision logic about single vs series editing to the controller. This will improve testability and make GUI code easier to maintain.

Third, there is Magic String smell for property names throughout the codebase. Property names like "subject", "start", "end" appear as strings in Calendar.java, EventEditor.java, and GUI dialogs. If someone misspells "descripton" instead of "description", the error only appears at runtime. To refactor this, create a PropertyName enum with values SUBJECT, START, END, DESCRIPTION, LOCATION, STATUS. Use this enum throughout the codebase instead of strings. This provides compile-time safety and makes property names discoverable through IDE autocomplete.

## C. Implementation Critique

The implementation has some solid parts. Using Java's modern time API like LocalDateTime and ZoneId makes date handling reliable and readable. EventRepository returns defensive copies with new ArrayList<>(events) in getAllEvents() method, which protects internal state from external modifications. The command parsing structure is clean and made adding new commands like dashboard easier because I could follow existing patterns.

But other parts are harder to maintain. EventRepository uses simple ArrayList for storage, which means operations like findEvent and getEventsOnDate have O(n) time complexity. For calendars with hundreds of events, these linear searches could become performance bottleneck. A better approach would be to use TreeMap or maintain multiple indexes by date and by subject for O(log n) lookups, though this increases memory usage. The current implementation prioritizes simplicity which is reasonable for current scale but limits scalability.

Error handling is inconsistent across different methods. Some methods return false when operations fail like removeEvent in Calendar, while others throw exceptions. EventOperationsHandler methods catch generic Exception and show error messages but return empty lists List.of() on failure, which masks errors from calling code. For example, if getEventsForDate fails, it returns empty list instead of propagating error, making debugging difficult. A more consistent approach would use checked exceptions for recoverable errors and unchecked exceptions for programming errors, with clear error messages throughout.

The GUI implementation mixes concerns in several places. Button listeners in dialogs directly call controller methods and then decide what to display based on results, which combines view and controller responsibilities. For instance, CreateEventDialog's create button handler validates input, calls controller methods, and shows success or error dialogs all in one method. This makes GUI code harder to test and maintain. The view should only handle display, with all business logic delegated to controller.

There is also a bug in the Edit Calendar functionality that affects the system. When trying to edit calendar properties, the code throws an error and does not work correctly. I informed the providers about this issue, but they could not provide a fix or explanation for it. This did not directly impact my ability to implement the dashboard feature, but it is one of the limitation in the existing codebase that should be noted.

The RecurringEvent class has performance issue in generateOccurrencesInRange method. It calls generateOccurrences() to create all occurrences first, then filters them by date range. For recurring event that repeats daily for 10 years, this generates over 3600 events even if user only wants events in one-month range. A better implementation would generate occurrences incrementally within the range, stopping once range is exceeded. This would reduce memory usage and improve response time for dashboard feature.

## D. Documentation Critique

The project includes helpful JavaDoc comments on most public methods and classes. Method signatures are clear and parameter descriptions are very informative. Files like design-changes.md and USEME.md provide useful context about architectural decisions and user instructions.

However, the most complex areas lack proper documentation. The GUI architecture lacks high-level overview explaining how dialog system works, how dialogs communicate with controller, and how different panels are composed. When implementing dashboard feature, it took time to understand DialogLauncher pattern and how to integrate new dialog consistently with existing ones. A README or architecture document in gui package explaining dialog lifecycle and communication patterns would help future developers.

The recurrence logic, while functional, lacks explanation of edge cases. For example, hasOccurrenceOn method in RecurringEvent calls generateOccurrences() when using repeatCount, which seems inefficient for simple check. There is no comment explaining why this approach was chosen or whether there is performance concern. Similarly, interaction between Calendar's series editing methods like editEvent, editEventsFromDate, editEntireSeries and how series IDs are managed could use comment block explaining series splitting logic.

The text interface command syntax is documented in USEME.md, but there is no quick reference guide. A commands.txt file exists in res directory, but it is not clear if this is comprehensive. A structured command reference table with syntax, examples, and error conditions would improve usability. Additionally, requirement that users must run use calendar --name X before most commands is mentioned but could be emphasized more prominently with examples showing what happens if you forget this step.

To address these limitations, I suggest: adding GUI_ARCHITECTURE.md document explaining dialog system and controller-view communication, adding comments to complex algorithms explaining approach and any performance considerations, creating COMMANDS.md quick reference with syntax tables and common error scenarios, and adding JavaDoc comments to private helper methods that contain non-obvious logic, especially in recurrence and series management code.
